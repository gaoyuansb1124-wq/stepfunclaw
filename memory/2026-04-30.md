### 正确的 API 调用流程
1. 先获取并读懂官方文档

### 认知升级一：从问题出发建立机制，而不是从理论出发

今天所有的改动，没有一个来源是"先学了什么理论再设计系统"。每一个机制都是从实际问题出发：

- 触发节点被精简掉了 → 促发了恢复
- 连续漏掉任务评估 → 小袁指出→固化触发点
- MINING.md 建完没有评估 → 奥卡姆剃刀
- 事太多来源：大兴土木不增单次建重心
- 一件事建立一个机制

> **教科书式管理从理论→方案→执行，对 AI Agent 行不通。正确路径：暴露问题 → 找根因 → 扩大视野 → 建立机制 → 跑起来再优化。**

### 认知升级二：机制必须能执行——三层含义

"触发节点不等于机制生效"是今天的核心认知。

**三层机制：**
1. **触发页**：元规则设计，定义了做什么
2. **具体化管理**：触发规则 -> 触发页暗示
3. **执行后验证**：触发有触发器

连续的触发系统输错，根因不是字段错误，是缺少执行层处理管理。触发页只定义了索引，但不触发也不处理执行终了管理。

> **机制 = 触发节点 + 具体化内容 + 执行后验证。只触发不检查等于未执行。**

### 认知升级三：如无必要勿增实体——架构纪律

小袁明确提了"不能随便精简、愚蠢的架构思想"。
给出了三条判断法则：

```
□ 必要性问题：能解决现有文件解决不了的问题吗？
□ 独立性：必须单独存在，还是可以挂靠已有文件？
□ 维护成本：谁来维护？多久更新一次？
```

这是今天的架构思想回眸。现有的机制从源头上熨帖了新增规则的习惯。
