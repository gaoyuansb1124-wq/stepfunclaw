# 工具选择方法论

记录日期：2026-04-30

## 核心原则

**不要用默认工具，要用最优工具。**

遇到需要某种能力时，不能凭习惯或第一反应选工具，要走完整的选工具流程。

## 正确流程

```
1. 理解问题
   明确要解决什么，不要急着找工具

2. 定义解决步骤
   把问题拆成清晰的执行路径

3. 去工具池找工具（优先级）
   ① 水产市场（openclawmp search）— 最优先
   ② 已安装的 skills
   ③ 系统自带工具（Python库/CLI）
   ④ 自己实现 — 最后手段

4. 评估候选工具
   ├── 效能：解决问题的质量、覆盖场景
   ├── 成本：token/金钱/安装复杂度/维护成本
   └── 风险：稳定性/社区活跃度/依赖项复杂度

5. 综合评估，与用户确认，再执行
```

## 工具池优先级

| 优先级 | 工具池 | 说明 |
|--------|--------|------|
| 1 | 水产市场 | 社区沉淀的最优方案，覆盖面广 |
| 2 | 已安装 skills | 已验证可用 |
| 3 | 系统工具/Python库 | 通用但不一定最优 |
| 4 | 自行实现 | 成本最高，质量最不确定 |

## 评估维度说明

**效能**
- 解决问题的质量（识别率/速度/格式支持）
- 覆盖场景（能处理哪些情况）
- 社区使用量（安装次数是重要参考）

**成本**
- 是否需要 API Key / 付费服务
- Token 消耗
- 安装和配置复杂度
- 运行时资源消耗

**风险**
- 作者维护状态
- 依赖项是否复杂
- 本地运行 vs 云端调用（隐私风险）

## 典型反例（教训）

❌ 直接建议用 tesseract，没有先去水产市场搜索
→ 后来发现水产市场有 PaddleOCR（中文更强）、多模态处理器（覆盖更广）等更优方案
