# 知识管理工作流

更新日期：2026-04-30

---

## 全局视角

```
一、知识采集（实时）
    原始材料（对话/文档/输入）
    → 打标签（[LRN][ERR][DEC][FEAT]）
    → 识别所属主题（对照 KNOWLEDGE-MAP.md）
    → 放入 knowledge/ 对应主题子目录

    ⚠️ 若主题不存在：
    → 判断是否需要新建主题
    → 是：先更新 KNOWLEDGE-MAP.md，再存放
    → 否：归入最近似的已有主题

二、知识沉淀（每日 cron 22:00）
    → 扫描当日 memory/YYYY-MM-DD.md
    → 提取带标签内容 [LRN][ERR][DEC][FEAT]
    → 对照 KNOWLEDGE-MAP.md 确认主题
    → 写入 knowledge/ 对应主题文件
    → 更新 heartbeat-state.json 的 lastKnowledgeExtract

三、知识成熟度评估（每日 cron 同步执行）
    → 检查 knowledge/ 各主题文件
    → 判断是否成熟（满足以下三条）：
      □ 经过多次实践验证
      □ 有具体案例支撑
      □ 逻辑自洽，可独立复用
    → 成熟 → 提炼进 assets/experiences/ 对应主题
    → 不成熟 → 继续迭代

四、知识图谱维护（每周）
    → 分析 KNOWLEDGE-MAP.md 主题分类
    → 判断是否需要细化/合并/优化
    → 更新 KNOWLEDGE-MAP.md

五、经验发布（按需）
    → 判断 assets/experiences/ 中经验是否成熟
    → 成熟 → 发布到水产市场
    → 不成熟 → 继续迭代
```

---

## 知识图谱（KNOWLEDGE-MAP.md）

知识库和经验库的主题目录，所有内容按主题归类存放。

**使用规则：**
- 存放知识前，先对照 KNOWLEDGE-MAP.md 确认主题
- 主题不存在时，先更新图谱再存放
- 定期优化主题分类，避免过于细碎或过于宽泛

---

## 与质检清单的关系

质检第6项触发本工作流的**第一步（知识采集）**：
```
□ 知识沉淀：本次对话有没有值得写进 knowledge/ 的知识点？
  → 有：对照 KNOWLEDGE-MAP.md 确认主题，立即写入
  → 没有：跳过
```
