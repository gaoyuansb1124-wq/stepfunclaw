# 踩坑：API 调用与接口对接

---

## PIT-101 | API 字段名猜错导致调用失败

**现象：** API 返回参数错误，反复修改 body 仍然失败

**根本原因：** 没有先读官方文档，凭印象猜字段名

**案例：** Get笔记 API
- 猜测用 `type`，实际是 `note_type`
- 猜测用 `url`，实际是 `link_url`

**正确做法：**
```
调用任何 API 前，必须先找到并读懂官方文档
确认字段名、数据类型、必填项后再写请求
```

**铁律：** 有文档先读文档，没文档先说没文档，不要用猜的

## PIT-102 | 出错后继续乱试而不找根因

**现象：** 第一次调用失败后，不断修改参数盲目重试

**根本原因：** 没有停下来找根本原因，靠运气试

**正确做法：**
```
第一次失败 → 停下来
→ 读错误信息（error.code / error.message）
→ 对照文档找原因
→ 理解后再试，一次解决
```

## PIT-103 | 安装 Skill 时 references 文件内容是自己编的

**现象：** Skill 安装后调用出错，原来 references 文件是推测写的

**根本原因：** 从 clawhub.ai 安装 skill 时，子页面抓不到，自己补写了 references，但字段名是猜的

**正确做法：**
1. 抓不到 references 时，明确告知用户："详细文档没拿到，以下内容是推测的，需要官方文档验证"
2. 用户提供官方文档后，先读透再更新 references
3. 调用前用真实文档验证一遍字段名

## PIT-104 | Get笔记异步任务 task_id 路径错误

**现象：** 保存链接成功但拿不到 task_id

**根本原因：** task_id 在 `data.tasks[0].task_id`，不是 `data.task_id`

**正确做法：**
```json
// 响应结构
{
  "data": {
    "tasks": [{"task_id": "xxx", "url": "..."}]
  }
}
// 取值：resp.data.tasks[0].task_id
```

## PIT-105 | Get笔记笔记详情数据路径错误

**根本原因：** 详情数据在 `data.note` 下，不是 `data` 直接取

**正确做法：** `resp.data.note.xxx`
