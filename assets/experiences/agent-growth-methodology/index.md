# 智能体养成方法论

_版本：2026-04-30 | 作者：小袁带教 | 状态：初稿_

---

## 一、核心认知

> **Agent 的成长不是从完善规则开始的，是从暴露问题开始的。**

教科书式的管理框架从头往后顺、一次性设计完全体系——对 AI Agent 行不通。因为：

1. **它不知道自己不知道什么** — 规则写多了反而嘈杂
2. **写进文件≠机制生效** — 机制必须嵌入触发节点才能执行
3. **提前想周全等于过度设计** — 实际跑起来之前，很多东西根本不知道怎么设计才对

**正确路径：问题驱动 → 从实践中提炼原则 → 再将原则沉淀为可执行规则 → 持续迭代。**

---

## 二、养成方法论：四步循环

```
发现实际问题
    ↓
找到根因（不是口号，是真问题）
    ↓
扩大视野（看更大的模式、看其他怎么解决）
    ↓
建立机制并嵌入流程（固定为触发节点）
    ↓
跑起来，再回来优化
```

**关键：不是"学理论→写方案→执行"，是"踩坑→知道疼了→找到原因→解决"。**

---

## 三、"机制必须能执行"的三层含义

**三层：** ① 机制要嵌入执行流程 ② 机制要嵌入文本文档 ③ 机制要具体化、可验证

以今天的实践为例：

| 「写了」 | 「没执行」 | 「 Reasons」 |
|---------|-----------|------------|
| AGENTS.md 写了任务评估触发点 | 连续6次漏掉 | 触发点只是文本描述，没有被强制触发 |
| AGENTS.md 写出了任务决策规则（有标准答案→S0，需分析→S1） | 该用S0的情况用了S1 | 判断准则太模糊，容易被实践冲淡 |
| AGENTS.md 写了 S1 触发节点 | S1 就会出现判断错误 | 决策节奏太快，模式的执行路径不清晰 |

**触发不是"你知道吗"而是"你能执行吗"。**

**机制 = 触发节点 + 具体语录 + 执行后被检查**。只触发不检查等于没有需求。

---

## 四、"从问题出发" vs "从理论出发"

| 从理论出发 | 从问题出发 |
|-----------|-----------|
| 先学习 CLAUDE.md 设计规则 | 先暴露实际问题 |
| 一次性地改变体系架构 | 每次只改那一个具体的规则 |
| 怕改错 | 非怕不改 |
| 不能优秀，想好再写 | 先写 PRACTICE 再做什么理论好 |
| "这个 AGTOS.mdM 应该多大" | "我忘触发了一个触发点" |

**本次实践：** 今天一天都在完成一个循环。先写 AGENTS.md → 写触发并发现问题 → 精选 CONSTRAINTS.md → 触发节点丢失 → 找回来 → 调试改写 → 触发丢失 → 写评审并新建 → 触发并失败 → 习惯 #7/#8 固化 → 系统复写。

## 五、生长的关键点

### 5.1  supervise 比 write 更有效

**不要动，先解释。** 暴露问题、解释思路、直到一目了然，最后才来写文件。

- 如果先写文件 → 别人要先读完、理解完才能参与
- 如果先讨论 → 双方同步、同步判断、同步推进

### 5.2  PTSD 从实践中来

- **问题驱动** — 每次刷新前先踩坑，踩坑后就记入问题库
- **每个坑 → 必须提出对应方案来解决**
- **而不是看到问题 → 事后修**

### 5.3 PEP 比 shrinkage 更重要

- PRD 要先有 BOUNDARY
- 简单一句话："什么是核心原则？"
- 复杂到可以写一个 PRD，流程later

### 5.4 推迟处理

如果今天一天能 6 个习惯固化、7 个执行规则，和 4 个机制 -- 明天还能更简化。

## 六、本文档的使用方式

- 每次遇到机制建设问题时，回来看第二节的循环
- 遇到"要不要新增文件/规则"时，回来看第四节对比表
- 养成方法论本身也要迭代——每完成一轮大的机制迭代，回来更新此文档
