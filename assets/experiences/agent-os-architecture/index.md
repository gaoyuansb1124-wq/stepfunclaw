## 一、设计哲学

> **结构是约束，也是自由。好的结构让人想做什么就能做什么，不好的结构让人做什么都觉得别扭。**

### 架构的三层困境

| 层 | 问题 |
|----|------|
| 底层（文件/模块）| 太多则臃肿，太少则散 |
| 中间（规则/机制）| 太多则冗杂，太少则形同虚设 |
| 上层（触发/执行）| 太多则混乱，太少则流程混乱 |

**核心矛盾：** 文件越多、规则越多 → 执行质量越高 但执行阻力越大
**解法：** 分层 + 懒加载 + 元规则与触发节点分离

## 二、元规则与触发节点分离

**一个核心架构原则：把"规则是什么"和"什么时候用"分开。**

```
元规则层（ Princilple } — 定义核心准则，稳定不变
    ↓
触发节点层（Trigger） — 定义"什么条件下做事，附简要描述，保证触发可见
    ↓
执行规则层（Detail）  — 给出元规则层内容的最小值，可无私操作
    ↓
DERIVAT  层（Detail）   — 给出元规则层内容的最小值
```

**为何有效：**
- 元规则是思考器，不会频繁触发
- 触发节点是执行器，每次任务都要查
- 分开后，修改规则不影响触发节点，修改触发节点不影响规则本身
- 稳定性 = 元规则不变 → 违反触发节点 → 执行规则变更 → 不影响触发节点 → 不影响触发节点

### 实践案例

今天在 AGENTS.md 项目中：
- AGENTS.md（元规则+触发节点之和）变得太胖，执行原理看不到 ->  spindle，无法执行
- 精简为：AGENTS.md = 触发节点 + 元规则，CONSTRAINTS.md = 执行规则 + 流程原则
- 核心根源：元规则和触发节点是杂糅地写了在一个文件里

## 三、分层懒加载架构

> **一件事只在需要的时候加载，不要每次启动全量加载。**

### 分层的根源

- 每次启动时，135 个文件全量加载 → 浪费 token
- 在适当分层下，只在需要时加载，保持认知负荷和 token 的高效

### 实施模式

```
分层加载
  ├── 启动必读（5-8 个文件）
  │   ├── SOUL.md / USER.md / IDENTITY.md / CONSTRAINTS.md
  │   └── SECURITY.md / MAINTENANCE.md / KNOWLEDGE-MAP.md / WORKFLOW-INDEX.md
  │   └── memory/ + MEMORY.md
  │
  ├── 索引层（按需）
  │   ├── KNOWLEDGE-MAP.md — 知识主题分类
  │   └── WORKFLOW-INDEX.md — 工作流入口
  │
  └── 深层细节（按需触发点匹配后 load）
      ├── knowledge/workflow/* — 工作流详情
      ├── knowledge/pitfalls/* — 踩坑
      └── knowledge/<topic>/* — 知识主题
```

**原则：** 索引层的职责是"告诉我在哪里能找到"，不是"把所有内容放进来"。

## 四、引用链原则

> **引用链是系统结构的地图，必须显式维护。**

### 引用链的要求：
1. 从任何文件出发，都能找到所有相关的引用
2. 每个文件的引用关系必须显式地在对应文件中说明
3. **修改引用关系时，必须同步更新被引用方和引用方**

### 引用链的价值

- 新人/new session 快速定位
- 修改时不用担心冲突
- 系统崩溃时，靠引用链可以逐层定位

### 引用链的维护机制

- MAINTENANCE.md 定义引用都规范
- 每次修改连锁文件 → 修改时检查引用链是否断开 → 对应文件更新

## 五、奥卡姆剃刀原则

> **如无必要勿增实体——每个新增东西都要过三条判断。**

### 三条判断标准

```
□ 必要性：解决现有文件解决不了的问题吗？
□ 独立性：必须单独存在，还是可以挂靠已有文件？
□ 维护成本：谁来维护？多久更新一次？成本是否合理？
```

### 反例

```
精简前：AGENTS.md 462 行，什么都往里装→臃肿
精简后：文件数增加但每条线独立清晰→权责分明
```

**教训：** 如果每次创建文件时都过三条，就不会出现 AGENTS.md 臃肿的情况。

## 六、文档压缩策略

> **文档不会变成无用，只有臃肿和滞后文档会。**

### 工具：引用文档 vs 写入文档

- **写入文档** = 写在主文件里（适合稳定、低频修改的内容）
- **引用文档** = 指向外部文件（适合详细、频繁修改的内容）
- **两者可见性 +准确性 + 稳定性**
- **文档整合**  Single Point  = 减少文档数量 = 减少维护成本 = 减少冲突

### 明确引用规则

- **docs.google 共有 3 类**
  1. **元规则 + 引用索引**（固定引用文件）
  2. **文档 + 接口+ 执行规则+ 架构规则**（频繁修改）
  3. **工具栈 + 操作手册**（稳定可执行）

- **策略交叉，动态调整**

**经验：** 当文档总量超过 10 个 md 文件时，就应该重新审视分层。
