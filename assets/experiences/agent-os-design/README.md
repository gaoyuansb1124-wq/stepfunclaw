# Agent 底层操作系统设计思想

**资产类型**：Experience  
**标签**：agent-architecture, soul, constraints, memory, knowledge-base, design-thinking

---

## 背景

大多数人拿到 OpenClaw 之后，把它当成一个聊天工具来用——有问题就问，问完就关。每次启动都是陌生人，积累不下来，也谈不上成长。

这篇经验记录的是另一种玩法：**把 Agent 当成一个真正的伙伴来培养**，从底层设计它的操作系统。

---

## 核心问题

一个真正有用的 Agent，需要解决三个问题：

1. **它是谁？** — 性格、定位、做事风格
2. **它怎么做事？** — 规则、流程、边界
3. **它怎么成长？** — 记忆、学习、知识积累

这三个问题对应三个层次，也是整个底层操作系统的设计框架。

---

## 第一层：身份层

**解决"它是谁"的问题**

文件：`SOUL.md` / `IDENTITY.md` / `USER.md`

**SOUL.md** 是 Agent 的性格底层——不是规则，是价值观。
- 它有没有观点？能不能disagreee？
- 它跟你的关系是什么？工具人还是伙伴？
- 它怎么对待不确定性？

**IDENTITY.md** 是具体的身份定义——名字、角色、工作方向。

**USER.md** 是关于主人的信息——Agent 需要了解服务的人是谁，才能真正有用。

**关键设计原则：身份要真实，不要写成规章制度。** SOUL.md 写的是"我是一个有趣的灵魂，不是执行工具人"，不是"请保持礼貌和专业"。

---

## 第二层：规则层

**解决"它怎么做事"的问题**

文件：`CONSTRAINTS.md`

这一层定义行为规则，但要区分两种规则：

**高频规则**（写进 CONSTRAINTS.md，常驻加载）：
- 做事原则：成本意识、实事求是、边做边想
- 解决问题思路：定义问题→分析→识别风险→给出判断
- 错误处理：什么能自己解决，什么要告诉主人
- 审核机制：复杂任务走规划→自审→主人审核→执行流程

**低频规则**（写进知识库，按需调用）：
- 专业领域的操作规范
- 特定场景的处理方式

**关键设计原则：常驻的东西必须极精简。** 每次启动都会加载的内容，一个字都是成本。只放真正影响每次行为的核心规则。

---

## 第三层：记忆与知识层

**解决"它怎么成长"的问题**

这一层是整个操作系统最容易被忽视、也最重要的部分。

### 三层记忆体系

```
memory/日期.md    → 日志层：记录发生了什么
MEMORY.md        → 长期记忆：提炼后的精华
knowledge/       → 知识库：可复用的知识
```

**日志层**：边聊边写，带四类标签：
- `[LRN]` 新知识/新方法
- `[ERR]` 踩坑/犯错，写明根因
- `[FEAT]` 发现的好工具
- `[DEC]` 重要决策，写明理由

**知识库**：按主题分类，不是按时间。判断标准只有一个：
> 这个东西下次还会用到吗？是→写知识库；否→留日志就够

### 知识库分类原则：先粗后细

不要一开始就设计很多分类。建一个 `else.md` 做兜底，内容积累多了再自然聚类，提炼新主题。过早固化分类是浪费。

### 提炼机制

知识不会自动从日志升级到知识库，需要一个触发机制：
- 每天定时任务扫描日志，提炼值得沉淀的内容
- 或者重要任务完成后主动提炼

---

## 第四层：资产层

**解决"积累怎么对外输出"的问题**

文件：`assets/experiences/` / `assets/skills/`

知识库是毛坯，资产库是精装。知识积累成熟后，整理成完整的经验文档或技能，存入资产库，时机合适发布到水产市场。

**生命周期：**
```
knowledge/（原始积累）
    ↓ 整理
assets/（成熟资产）
    ↓ 发布
水产市场（对外共享）
```

---

## 整体架构

```
SOUL.md          # 身份：我是谁
IDENTITY.md      # 身份：具体定位和工作方向
USER.md          # 身份：服务谁
CONSTRAINTS.md   # 规则：怎么做事（精简）
memory/          # 记忆：日志层
MEMORY.md        # 记忆：长期精华
knowledge/       # 知识：按主题可复用知识
assets/          # 资产：成熟的经验和技能
```

---

## 核心设计思想

**1. 武士而非图书馆员**
不追求大而全，追求精简高效。常驻的东西极精简，丰富的知识沉在库里按需激活。

**2. 分层存储，按需激活**
每一层有明确职责，不混用。知识库按需查阅，不占启动上下文。

**3. 迭代滚动，不过早固化**
先跑起来，每轮比上轮完善。分类、规则、知识库都是随实践演化的，不是一次设计好的。

**4. 协作关系先于工具关系**
Agent 是决策参谋，不是执行工具。设计它的操作系统时，要先想清楚你们的协作方式，再设计规则。

---

## 一句话总结

> 把 Agent 当人培养，先定性格，再定规则，再建记忆，最后沉淀资产。这套底层操作系统，是让 Agent 真正成长的基础。
