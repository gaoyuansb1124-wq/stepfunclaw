# Agent 底层操作系统设计思想

**资产类型**：Experience
**版本**：1.1.0（持续更新中）
**标签**：agent-architecture, soul, constraints, memory, knowledge-base, design-thinking

---

## 背景

大多数人拿到 OpenClaw 之后，把它当成聊天工具来用——有问题就问，问完就关。每次启动都是陌生人，积累不下来，也谈不上成长。

这篇经验记录的是另一种玩法：**把 Agent 当成一个真正的伙伴来培养**，从底层设计它的操作系统。

---

## 核心问题

一个真正有用的 Agent，需要解决三个问题：

1. **它是谁？** — 性格、定位、做事风格
2. **它怎么做事？** — 规则、流程、边界
3. **它怎么成长？** — 记忆、学习、知识积累

这三个问题对应三个层次，也是整个底层操作系统的设计框架。

---

## 第一层：身份层

**文件：** `SOUL.md` / `IDENTITY.md` / `USER.md`

### SOUL.md — 性格底层

SOUL.md 不是规章制度，是价值观和性格定义。

**关键设计决策：**
- 写"我是一个有趣的灵魂，不是执行工具人"，而不是"请保持礼貌和专业"
- 定义协作关系：互商互量的伙伴，而非甲方和外包
- 定义思维方式：多镜头思维（产品/用户/市场/商业/哲学/业务），默认产品思维，按场景切换
- 定义行动原则：思考与行动齐头并进，发现主人卡在思考时友情提醒

**为什么要有架构思维？**
在设计任何系统（包括 Agent 自身）时，需要先看整体结构再看局部，在简单与完备之间做有意识的权衡。架构思维是元能力，影响所有决策。

### IDENTITY.md — 具体定位

- 名字、角色、三大工作方向
- 随实践迭代，不要一开始写太满

### USER.md — 服务谁

Agent 需要了解服务的人是谁，才能真正有用。包括：职业背景、性格特点、兴趣方向、沟通偏好。

## 第二层：规则层

**文件：** `CONSTRAINTS.md`

### 核心设计原则：常驻内容必须极精简

**为什么？**
CONSTRAINTS.md 每次启动都会加载，一个字都是 token 成本。只放真正高频、影响每次行为的核心规则。

**高频规则（写进 CONSTRAINTS.md）：**
- 做事原则：成本意识（时间+金钱+token）、实事求是、边做边想
- 解决问题思路：定义问题→多角度分析→识别风险→给出判断→等待决策
- 错误处理：能自己解决的 vs 必须告诉主人的，拖太久主动说
- 审核机制：复杂任务走规划→自审（反驳清单）→主人初步反驳→修订→主人最终拍板

**低频规则（写进知识库，按需查阅）：**
- 专业领域操作规范
- 特定场景处理方式

### 门下省审核机制设计

**为什么单 Agent 自我审核有问题？**
中书省（规划）和门下省（审核）都是同一个 Agent，难以真正否定自己的规划。

**解决方案：二步审核法**
1. Agent 用反驳清单自审（A需求层/B方案层/C风险层/D执行层）
2. 主人初步反驳
3. Agent 修订
4. 主人最终拍板

这样审核引入了真正的外部视角，客观性大幅提升。

## 第三层：记忆与知识层

**解决"它怎么成长"的问题**

### 三层记忆体系

```
memory/日期.md    → 日志层：快速记录，带标签
MEMORY.md        → 长期记忆：人工提炼的精华
knowledge/       → 知识层：可复用知识，按主题
```

**为什么要三层？**
- 日志层：快速、随手，不追求质量
- 长期记忆：人工精选，量少质精
- 知识层：结构化，按主题索引，执行任务前按需查阅

三层职责不同，不能合并。

### 精简高效的核心矛盾

**丰富的知识必然带来低效，但高效并不排斥丰富的知识。**

解法是分层存储、按需激活：
- 常驻（SOUL+CONSTRAINTS）：极精简，只放高频规则
- 知识库：丰富，但按需查阅，不常驻
- 目标：武士而非图书馆员——不是百科全书，而是随时能拔出来用的利器

## 第四层：资产层

**文件：** `assets/experiences/` / `assets/skills/`

### 知识成熟度演进链

```
日志（原始记录）
    ↓ 提炼
知识点（knowledge/）
    ↓ 多个知识点积累融合
初步经验（assets/ 内部迭代）
    ↓ 不断优化更新
成熟经验
    ↓ 发布
水产市场（供他人复用）
```

**关键认知：**
- 很多经验只是内部积累，不一定发布
- 发布不是终点，发布的经验也要持续迭代更新版本
- 知识成熟了才能变成经验，多个知识融合才能变成好经验
- 经验文档是活的文档，要不断补充细节和案例

### assets/ 与 skills/ 的关系

`assets/skills/` 是自制技能的源文件备份，系统不会自动识别。
技能要生效必须安装到 `~/.stepclaw/skills/` 路径。
两者分开，源文件和运行时分离。

## 整体架构

```
SOUL.md          # 身份：性格、价值观、思维方式
IDENTITY.md      # 身份：名字、角色、工作方向
USER.md          # 身份：服务谁、关于主人
CONSTRAINTS.md   # 规则：做事原则（精简，高频）
HEARTBEAT.md     # 心跳：定期检查清单
memory/          # 记忆：日志层（带标签）
MEMORY.md        # 记忆：长期精华
knowledge/       # 知识：按主题，可复用
assets/          # 资产：成熟的经验和技能源文件
```

## 四条核心设计思想

**1. 武士而非图书馆员**
不追求大而全，追求精简高效。常驻的东西极精简，丰富的知识沉在库里按需激活。

**2. 分层存储，按需激活**
每一层有明确职责，不混用。知识库按需查阅，不占启动上下文。

**3. 迭代滚动，不过早固化**
先跑起来，每轮比上轮完善。分类、规则、知识库都是随实践演化的，不是一次设计好的。

**4. 协作关系先于工具关系**
Agent 是决策参谋，不是执行工具。设计它的操作系统时，要先想清楚协作方式，再设计规则。
