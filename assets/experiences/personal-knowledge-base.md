## 背景与问题

AI Agent 每次启动都是空白的，记忆靠外部文件。市面上有两类方案：

**方案一：知识库类**（如 learning-knowledge-base）
- 思路：图书馆式，按主题分类存档，只增不减
- 盲点：只管存，不管怎么产生。知识从哪来？靠人工塞，没有自动生成机制

**方案二：日志记录类**（如 learn-journal）
- 思路：日记本式，按事件类型结构化记录
- 盲点：只管记，没有提炼沉淀机制。日志越积越多，但怎么变成可复用的知识？没有答案

**关键洞察**：两者是上下游关系，不是竞争关系。日志记录是原材料，知识库是成品，中间缺一个提炼机制把两者串联起来。

---

## 核心设计：两层架构

```
对话/任务中发生事件
        ↓ 边发生边记录（不攒到最后）
【日志层】memory/日期.md
        ↓ 定期提炼（心跳/复盘触发）
【知识层】knowledge/主题文件
        ↓ 执行任务前按需查阅
```

**为什么是两层而不是一层？**
- 日志需要快速、随手记，格式可以松散
- 知识需要结构化、可复用，需要整理
- 强行合并会导致两头都做不好

## 日志层设计

### 文件位置
`memory/YYYY-MM-DD.md`，按日期存放

### 四类标签

| 标签 | 含义 | 何时用 |
|------|------|--------|
| `[LRN]` | 新知识/新方法 | 学到了可复用的东西 |
| `[ERR]` | 踩坑/犯错 | 出了问题，写明根因和预防 |
| `[FEAT]` | 发现好工具/资产 | 找到有价值的外部资源 |
| `[DEC]` | 重要决策 | 做了关键选择，写明为什么 |

### 关键原则：边聊边写，不攒到最后

**为什么不能攒到最后？**
Agent 没有"对话结束"的感知信号——用户可能直接关窗口，Agent 来不及写。
所以必须养成"发现就记"的习惯，不依赖对话结束时的集中写入。

## 知识层设计

### 文件结构
```
knowledge/
├── INDEX.md         # 目录 + 分类规则
├── thinkmodel.md    # 思维模型与分析方法论
├── product.md       # 产品经理相关
├── buddhism.md      # 佛学相关
├── business.md      # 创业/业务相关
├── agent-growth.md  # 智能体养成
└── else.md          # 其他兜底
```

### 分类原则：先粗后细

**为什么要有 else.md 兜底？**
过早固化分类是浪费。刚开始内容少，不知道哪些主题会积累出来。
先把归不了类的放 else.md，等积累到一定量，再扫描聚类、提炼新主题、迁移内容。

**判断标准：**
> 这个东西下次还会用到吗？
> - 是 → 写进 knowledge/ 对应主题文件
> - 否 → 留在日志就够

### 知识 vs 教训的区别

| 放知识库 | 放日志 |
|---------|--------|
| 可复用的方法、框架 | 踩坑记录、错误教训 |
| 成功的经验模式 | 过程日志、临时记录 |
| 领域知识精华 | 失败教训 |

## 提炼机制

**触发时机：**
- 定时任务（每天22:00）自动扫描当天日志
- 重要任务完成后主动提炼
- 心跳检测：今天未提炼则触发补跑

**提炼流程：**
1. 扫描日志中带标签的内容
2. 判断是否值得进知识库（下次还会用到？）
3. 确定归属主题（product/buddhism/business/agent-growth/thinkmodel/else）
4. 写入对应文件
5. 更新 heartbeat-state.json 记录提炼时间

## 与其他记忆层的关系

| 文件 | 定位 | 特点 |
|------|------|------|
| `memory/日期.md` | 日志层，原始记录 | 快速、随手、带标签 |
| `MEMORY.md` | 长期记忆精华 | 人工提炼的最重要内容 |
| `knowledge/` | 知识层，可复用知识 | 按主题、结构化、持续积累 |

三层不重叠，各司其职。

## 流程可行性质检

| 环节 | 风险 | 解决方式 |
|------|------|---------|
| 识别值得记录的内容 | 可能漏掉隐性知识 | 主动识别习惯 |
| 写入时机 | 对话结束来不及写 | **边聊边写**，不攒到最后 |
| 提炼标准 | 判断主观，初期不稳定 | 主人抽查反馈，校准标准 |
| 任务前查阅 | 初期可能忘 | 写入 CONSTRAINTS.md 成规范 |

## 已知不足与演进方向

- **知识老化**：存进去的知识可能随时间过期，暂时接受，未来配置 cron 定期巡检
- **语义搜索**：知识积累多了之后，关键词查找效率下降，届时引入 memsearch
- **跨 session 一致性**：每次启动不可能把所有知识库都读一遍，靠按需查阅解决
