# 实操案例：AGENTS.md 精简与触发节点重建

_版本：2026-04-30 | 场景：核心文件体系重构 | 状态：完结_

---

## 一、问题

AGENTS.md 初始版本 462 行 / 18KB，包含：
- 系统配置 ✅（应在）
- 启动序列 ✅（应在）
- 安全规则一~六章 ❌（不应在这里）
- 工作流/文档处理细节 ❌（不应在这里）
- 浏览器经验 ❌（不应在这里）
- 三省六部完整规则 ❌（不应在这里）

**问题：AGENTS.md 每次启动都要全量加载，重重赘述导致规则稀释、执行忽略关键信息。**

---

## 二、错误路径：一次性精简

**做了什么：** 一股脑拆分 → 把安全规则六章迁移到 SECURITY.md → 把触发逻辑概要留在 AGENTS.md → 删除详细规则

**结果：**
```
AGENTS.md（导入 140 行）
  → CONSTRAINTS.md（触发节点详细内容）
  → SECURITY.md（安全规则）

**问题：触发节点在 AGENTS.md 只有概要，不在 CONSTRAINTS.md 被覆盖了**
```

**后果：** 后续6次连续漏掉任务复杂度评估，触发机制形同虚设。

**失败根源：** 一次改动多个文件、多个维度、多个层面，导致无法判断是哪一步出了问题。

---

## 三、修复路径：逐步重建

### 第一步：发现触发节点丢失

小袁提醒"强制执行逻辑好像没有了"→ 确认 AGENTS.md 精简后确实删了 S0/S1/质检清单的详细内容。

**根因：** 把触发器从 AGENTS.md 移走时，只留了概要，没有同步把详细内容放到 CONSTRAINTS.md，导致执行时加载了 AGENTS.md 概要，但详细规则丢失。

### 第二步：恢复触发节点到 CONSTRAINTS.md

- S0 白名单详细列表 → CONSTRAINTS.md
- S1 五维打分表 → CONSTRAINTS.md
- 六项质检清单（含知识沉淀）→ CONSTRAINTS.md
- "有标准答案→S0，需推理→S1"判断原则 → CONSTRAINTS.md

### 第三步：AGENTS.md 加入兜底声明

```
If CONSTRAINTS.md fails to load, the summary above is self-sufficient.
Do not skip execution just because the detailed file is unavailable.
```

### 第四步：新增"任务完成后复盘"触发节点

从"单行提醒"升级为 6 维度清单（新知识点/新工作方法/坑总结/经验规律/对用户的认识/养成案例）。

### 第五步：强制评审流程写入 Pre-task

修改核心文件 → 必须提交修改建议 → 用户确认 → 才能执行 → 确保 git 可回退。

### 第六步：习惯 #7/#8 固化

- #7 同路试三次，不成就换路（止损机制）
- #8 结构变更一次只搞一个（防止一次改太多）

---

## 四、架构原则

### 4.1 元规则与触发节点分离

```
AGENTS.md（触发节点 + 元规则概要） ← 稳定不变
CONSTRAINTS.md（执行规则 + 流程细节） ← 随实践迭代
```

**两者必须同时加载才能生效。** 如果 AGENTS.md 失效但 CONSTRAINTS.md 存在，AGENTS.md 的"必须加载 CONSTRAINTS.md"指令反而能兜底。

### 4.2 引用链可用性验证

每改动一次引用链，都应该验证完整性：
```
9 个启动文件 → 全部存在？
16 个索引文件 → 全部存在？
```

### 4.3 触发节点≠机制生效

触发节点写在文件里，但必须：
- **嵌入执行流程** — 直接嵌入 Pre-task / Post-task 检查点
- **有监督机制** — 小袁的持续监督比任何文档都有用
- **可验证** — 每个触发节点都应有可检查的输出

### 4.4 结构变更优先于内容

结构变更（文件拆分/合并/新建/删除）必须：
- 一次只做一种变更
- 单独确认后继续
- 绝不同时改结构 + 写内容 + 推送

---

## 五、关键教训总结

| 教训 | 对应机制 |
|------|---------|
| 精简文件时不能只删不补 | MAINTENANCE.md → 修改前检查5步 |
| 机制写进文件≠机制生效 | CONSTRAINTS.md → 触发节点嵌入流程 |
| 触发节点只有概要会导致执行失败 | 触发节点+执行规则双路径 |
| 一次改多个文件导致问题难以定位 | 习惯 #8：结构变更一次只搞一个 |
| 没有"新增评估"导致文件膨胀 | MAINTENANCE.md → 奥卡姆三条 |
| 连续失败不换路是在浪费时间 | 习惯 #7：同路试三次换思路 |

## 六、本文档的使用方式

- 遇到文件体系重构时，用第三节的"错误路径 vs 修复路径"对照
- 遇到"精简还是扩建"时，用第四节架构原则决策
- 每次完成机制迭代后，回来更新本节关键教训
