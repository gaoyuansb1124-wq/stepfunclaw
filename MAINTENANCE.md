## 二、修改前检查流程

修改任何核心文件前，必须执行以下检查：

```
1. 明确修改目的：为什么改？改什么？
2. 对照本文件，确认修改符合该文件的职责定位
3. 检查是否与其他核心文件有内容重叠/冲突
4. 检查引用链是否完整（修改后是否有文件指向不存在的内容）
5. 如有冲突或不确定 → 告知用户，等待决策
```

### 新增文件前评估（奥卡姆剃刀原则：如无必要勿增实体）

**新建任何文件前，必须过以下三条：**

```
□ 必要性问题：这个文件能解决一个现有文件解决不了的问题吗？
  → 否 → 考虑合并进已有文件，不要新建
□ 独立性：这个文件的内容必须单独存在，还是可以挂靠已有文件？
  → 可以挂靠 → 合并，不要新建
□ 维护成本：谁来维护？多久更新一次？维护成本是否合理？
  → 维护成本不合理 → 重新评估是否真的需要
```

**三条全过 → 可以新建。任何一条没过 → 先考虑合并/挂靠。**

> 如无必要勿增实体。每增加一个文件，都增加了系统的维护负担和认知消耗。
> 不在万不得已时，不动结构的底层。

## 三、文件间引用关系

```
AGENTS.md（总入口）
  → CONSTRAINTS.md（做事原则）
  → SECURITY.md（安全规则）
  → MAINTENANCE.md（维护规则）
  → WORKFLOW-INDEX.md（工作流索引）
  → KNOWLEDGE-MAP.md（知识图谱）
  → SOUL.md / USER.md / IDENTITY.md（身份）
  → MEMORY.md（长期记忆）

CONSTRAINTS.md
  → 引用 KNOWLEDGE-MAP.md（知识主题分类）
  → 引用 WORKFLOW-INDEX.md（工作流）

WORKFLOW-INDEX.md
  → 引用 knowledge/workflow/*.md（详细工作流）
  → 引用 skills/getnote/references/*.md（API参考）

KNOWLEDGE-MAP.md
  → 定义 knowledge/ 主题分类
  → 定义 assets/experiences/ 经验分类
```

**修改规则：** 修改引用关系时，必须同步更新被引用方和引用方。

## 四、认知升级固化路由

认知升级（思维框架/理念）和执行规则（操作流程）的固化路由：

```
认知升级（思维框架/设计理念） → IDENTITY.md  design_principles 节
执行规则（流程/习惯/触发节点）  → CONSTRAINTS.md 对应章节
具体知识点/方法论             → knowledge/ 或 assets/experiences/
长期记忆                     → MEMORY.md
```

**触发轮次：** 每次任务结束后的六维复盘，其中一个维度"认知升级检查"自动触发固化。

**固化顺序：** 先路线，后固化写入，按以下顺序执行：
1. 识别认知类型 → 思维框架 / 执行规则 / 具体知识
2. 按路由写入对应文件
3. 完成后返回下一步骤

**固化前检查：**
```
□ 对应文件是否存在？不存在 → 先新增
□ 是否与现有内容冲突？有冲突 → 先解决冲突
□ 路由选择是否合理？不合理 → 重新评估
```

---

## 五、冲突处理机制

**当修改内容与维护规则冲突时：**

1. 停止修改
2. 告知用户：「修改 X 与维护规则 Y 冲突，具体是...」
3. 用户判定：
   - 不需要改维护规则 → 按原规则执行，不修改
   - 需要更新维护规则 → 先更新 MAINTENANCE.md，再执行修改
4. 修改完成后，同步检查 MAINTENANCE.md 是否需要更新
